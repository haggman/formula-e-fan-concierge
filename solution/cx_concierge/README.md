# CX Concierge — reference build (Conversational Agents / CX Agent Studio)

The **pull** surface: an embedded chat that answers anything a fan asks. It is an
**orchestrator** — it owns no data itself; it routes each question to a grounded tool and
answers only from what that tool returns.

This package is mostly **configuration + grounding assets**, because CX Agent Studio is
low-code. The "developer meat" is the wire to the race-data subagent, the RAG grounding,
the auth, and the routing playbook. The app definition is built in the console and exported
here (`exportApp`/`importApp`) so it lives in version control.

## Tool surface (resolved by the integration spike — `spec/cx_integration_spike.md`)

| Need | CX tool | Setup |
|---|---|---|
| Live race + in-race stats ("how's car 13 now?", standings, "Evans' recent seasons?") | **OpenAPI tool** `ask_race_data` → the race-data subagent on Cloud Run (Service Agent ID Token; the agent IS the service) | `../race_data_subagent/CX_SETUP.md` |
| Driver & team **profiles** | **Data store tool** (RAG) over the **Driver & Team Profiles** store | `DATASTORE_SETUP.md` |
| **Rules** ("what's Attack Mode?") | **Data store tool** (RAG) over the **FE Rules** store (FIA PDFs + concise pack) | `DATASTORE_SETUP.md` |
| Current / open-web | **Google Search tool** (grounding) | add in CX |

> **Transport note:** the spike chose the **OpenAPI tool**, not MCP. CX has a first-class
> OpenAPI tool, the single `ask_race_data` operation maps to CX's one-operation rule, and
> same-project Cloud Run auth is trivial. MCP-on-Cloud-Run also works but adds the MCP
> mental model for no student payoff; A2A/Agent-Registry aren't consumable by CX. Full
> reasoning + the validated wire are in `spec/cx_integration_spike.md`.

Not the backbone: **Agent-as-a-tool** (in-app only, not our external ADK subagent),
**MCP tool** (works, not chosen), **Integration Connector** (possible direct-Firestore
contrast/stretch).

## The orchestration playbook

`AGENT_PLAYBOOK.md` is the agent's **goal + routing instructions + examples** to paste into
CX. The core discipline (the spike's lesson): answer **only** from tool results — with an
unguided prompt the orchestrator invents positions/facts from training data — and stay
**time-honest**: relay the subagent's "who wins?" refusal, never spoil the replay.

## Grounding assets

See `grounding/README.md`. The rules pack is authored (the dataset has no rules/bio corpus);
the driver/team profiles are generated by `grounding/build_profiles.ipynb` (comprehensive —
all FE drivers + teams), grounded in the career/results tables and verified five ways. Both
are staged to `gs://class-demo/formula-e/grounding/` as `.txt`; students build the two
Vertex AI Search data stores from there (`DATASTORE_SETUP.md`).
